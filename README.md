> [!TIP]
> Remember, I used a combination of Visual Studio references and GitHub Sub Modules to ensure you can tackle discovery, either at the highest enterprise-project-level of the Primary JSopX™ Open Project EXperiences Collection or by accessing Projects independently.

Using a combination of Visual Studio references and GitHub Sub Modules has its pros and cons. Yes, easier for you to do discovery at any level (by design). Conversely, this approach makes how you clone repositories a little differently, and it is also a little harder to manage projects, when and if you use more common approaches you find on the web vs how we need to handle the same operations, but subtle, but noticable ways. You will want to consider a mono-repo approach in your versions of your enterprise-level, production ready applications for easier, top-level, all-in-one, project repository management if you find GitHub Sub Modules a little daunting. At the end of the day, it is about the business requirements of the collective projects. In our case, the business requirement for this release of projects was to allow you free reign to explore any or all projects in bite sizes you can most comfortably ingest, regardless of any growing pains me personally, the quote on quote, "developer" endures as a result of that business requirement.

We will use the `Installation` process as an example.

### Default Installation

Typical Repository Solution with all projects under one roof:

1. **Clone the repository**:

    ```bash
    git clone https://github.com/JasonSilvestri/JSopX.BridgeTooFar.git
    ```

2. **Open the solution in Visual Studio**:

    - Open `JSopX.BridgeTooFar.sln` in Visual Studio.

3. **Build the project**:

    - Right-click on the solution in Solution Explorer and select `Build Solution`.


### Our Custom Installation

Custom Repository Solution with all projects under one roof and/or giving you the ability to discovery projects independently:

1. **Clone the repository, while ensuring we have properly updated sub modules**:

    ```bash
    git clone --recurse-submodules https://github.com/JasonSilvestri/JSopX.BridgeTooFar.git
    ```

2. **Open the solution in Visual Studio**:

    - Open `JSopX.BridgeTooFar.sln` in Visual Studio.


3. **Build the project**:

    - Right-click on the solution in Solution Explorer and select `Build Solution`.

### Our Custom Installation with Special Considerations

As you can see above, there isn't a huge difference in cloning the repository of choice. This is because much of what needs to be done initially I have done for us. 

Still, there are some special considerations, in special cases, that may or may not need to be executed, due to a few factors. 

1. **Clone the repository, while ensuring we have properly updated sub modules**:

    ```bash
    git clone --recurse-submodules https://github.com/JasonSilvestri/JSopX.BridgeTooFar.git
    ```

2. Sometimes, navigating to the Project Directory and ensuring Sub Modules are initialized and updated is a need:

    ```bash
    cd localcomputer/repos/JSopX.BridgeTooFar
    ```

3. Initialize and Update Submodules:

    ```bash
    git submodule update --init --recursive
    ```

#### Other Special Considerations

4. Keeping Submodules Up-to-Date

Submodules do not automatically update when you pull changes from the main repository. You need to update them manually to match the commits referenced in the parent repository.
Checking Submodule Status

To see the status of your submodules:

    ```bash
    git submodule status
    ```

This command shows the current commit of each submodule.

5. Updating Submodules

After pulling changes from the main repository, you might need to update the submodules:

    ```bash
    git submodule update --recursive 
    ```

6. **Open the solution in Visual Studio**:

    - Open `JSopX.BridgeTooFar.sln` in Visual Studio.


7. **Build the project**:

    - Right-click on the solution in Solution Explorer and select `Build Solution`.
